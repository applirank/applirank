# Authentication

Reqcore uses [Better Auth](https://www.better-auth.com) for authentication and the Better Auth Organization plugin for multi-tenancy.

## How It Works

```
Sign Up → Create Account → Create Organization → Dashboard
Sign In → Session Cookie → Auth Middleware → Access Dashboard
```

Better Auth handles:

- **Email/password** registration and login
- **Session management** with HTTP-only cookies
- **Organization membership** with role-based access
- **OAuth providers** (configurable)

## Server-Side Auth

### Auth Guard

Every authenticated API route calls `requireAuth(event)`:

```ts
// server/api/jobs/index.get.ts
export default defineEventHandler(async (event) => {
  const { session, orgId } = await requireAuth(event)
  // session is guaranteed valid, orgId is the active organization
  const jobs = await db.select().from(schema.job)
    .where(eq(schema.job.organizationId, orgId))
  return jobs
})
```

The `requireAuth` utility:

1. Validates the session cookie
2. Extracts the active organization ID
3. Throws `401` if not authenticated
4. Throws `403` if no active organization

### Auth API Route

Better Auth exposes a catch-all route at `server/api/auth/[...all].ts` that handles all auth-related endpoints (sign-up, sign-in, sign-out, session, etc.).

## Client-Side Auth

### Auth Client

The Better Auth Vue client is configured in `app/utils/auth-client.ts` and is auto-imported:

```vue
<script setup>
const { data: session } = await authClient.useSession(useFetch)
</script>
```

### Middleware

Reqcore uses three route middleware:

| Middleware | Purpose |
|-----------|---------|
| `auth` | Redirects unauthenticated users to sign-in |
| `guest` | Redirects authenticated users away from auth pages |
| `require-org` | Redirects users without an active organization to onboarding |

### SSR Cookie Forwarding

During server-side rendering, cookies are not automatically forwarded. All authenticated `useFetch` calls include:

```ts
const { data } = await useFetch('/api/protected', {
  headers: useRequestHeaders(['cookie']),
})
```

## Multi-Tenancy

### Organizations

Each user can belong to multiple organizations. The active organization is stored in the session. All data queries are scoped by the active organization's ID.

### Roles

| Role | Capabilities |
|------|-------------|
| **Owner** | Full access, manage members, delete organization |
| **Admin** | Manage jobs, candidates, applications, settings |
| **Member** | View and manage assigned jobs and candidates |

### Switching Organizations

Users can switch between organizations using the organization switcher in the sidebar. Switching updates the `activeOrganizationId` in the session.

## Onboarding Flow

1. User signs up (creates account)
2. `require-org` middleware detects no active organization
3. User is redirected to `/onboarding/create-org`
4. User creates their first organization
5. User is redirected to the dashboard
