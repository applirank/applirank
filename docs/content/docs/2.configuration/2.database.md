# Database

Reqcore uses **PostgreSQL 16** as its primary database, accessed through **Drizzle ORM** with the `postgres.js` driver.

## Connection

The database connection is configured via the `DATABASE_URL` environment variable:

```
DATABASE_URL=postgresql://user:password@localhost:5432/reqcore
```

The connection pool is managed in `server/utils/db.ts` and is auto-imported across all server code.

## Schema

Database tables are defined in `server/database/schema/`:

| File | Tables | Purpose |
|------|--------|---------|
| `app.ts` | `job`, `candidate`, `application`, `document`, `question`, `answer` | Domain tables |
| `auth.ts` | `user`, `session`, `account`, `organization`, `member` | Better Auth tables (DO NOT MODIFY) |
| `index.ts` | — | Re-exports all schemas |

### Data Model

```
organization (Better Auth)
├── job (draft → open → closed → archived)
│   └── application (new → screening → interview → offer → hired/rejected)
│       └── candidate
│           └── document (resume, cover_letter — stored in S3)
└── member (user ↔ organization with role)
```

All domain tables have an `organizationId` foreign key for tenant isolation.

## Migrations

### Auto-Apply on Startup

The `server/plugins/migrations.ts` plugin runs Drizzle migrations automatically when the server starts. It uses a PostgreSQL advisory lock (`pg_try_advisory_lock`) to prevent race conditions in multi-instance deployments.

### Generate Migrations

After modifying a schema file, generate a migration:

```bash
npm run db:generate
```

This creates a new SQL migration file in `server/database/migrations/`.

### Apply Migrations Manually

```bash
npm run db:migrate
```

### Other Commands

| Command | Description |
|---------|-------------|
| `npm run db:push` | Push schema directly (development only — skips migrations) |
| `npm run db:studio` | Open Drizzle Studio (visual database browser) |
| `npm run db:seed` | Seed the database with demo data |

## Multi-Tenancy

Every database query is scoped by `organizationId`, derived from the authenticated session:

```ts
// Example: list jobs for the current organization
const jobs = await db
  .select()
  .from(schema.job)
  .where(eq(schema.job.organizationId, orgId))
```

::callout{icon="i-lucide-shield-alert" color="error"}
**Security invariant**: The `organizationId` is ALWAYS extracted from the authenticated session, NEVER from user input (body, query, URL params). This is the #1 security rule in Reqcore.
::

## Candidate Deduplication

Candidates are deduplicated within each organization by email using a composite unique index:

```
uniqueIndex(organizationId, email)
```

This means the same person can exist as a candidate in different organizations, but each organization can only have one candidate record per email address.
